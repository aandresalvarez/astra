## Phase 0 ‚Äî Account and asset setup (one-time, no code)

1. **Create a Developer ID Application certificate** in the Apple Developer portal (Account Holder role required ‚Äî fine for a personal enrollment). Record the Team ID. Do **not** create a Developer ID Installer cert yet ‚Äî distribution is a zip, not a `.pkg`.
2. **Export and back up the private key** (`.p12` with a strong passphrase) to a password manager. This key can only be created a limited number of times and losing it complicates CI setup later.
3. **Store notary credentials:** create an **App Store Connect API key** (preferred over app-specific password ‚Äî it also works headless in CI later) and run
   `xcrun notarytool store-credentials astra-notary --key <key.p8> --key-id <id> --issuer <issuer>`.
   The profile name `astra-notary` becomes `ASTRA_NOTARY_PROFILE`.
4. **Verify identity resolution:** `security find-identity -v -p codesigning` shows `Developer ID Application: <name> (<TEAMID>)`. That exact string is `ASTRA_SIGN_IDENTITY`.

Deliverable: nothing in-repo; a short "assets checklist" appended to `docs/code-signing.md` (cert name, Team ID, notary profile name, where the `.p12` backup lives ‚Äî **never** the secrets themselves).

### 1a. Replace `--deep` with explicit inside-out signing (developer-id path)

`codesign --deep` is deprecated for distribution and has two concrete failure modes here:
- It stamps the **app's entitlements onto every nested binary**, which breaks Sparkle's `Autoupdate`/`Updater.app`/XPC helpers (they need their own entitlements, or Sparkle's own valid signature preserved).
- It signs nested items with default flags in ways notarization can reject.

Change the signing step (currently `build_and_run.sh:305‚Äì316`) so that when a real identity is set it signs inside-out:
1. Each of the 7 bundled tools (`astra-browser`, `astra-mcp-gateway`, `astra-host-control`, `astra-workspace`, `stanford-mail`, `stanford-apple-mail`, `stanford-graph-mail`) individually with `--options runtime --timestamp`, **no** app entitlements.
2. Sparkle.framework's nested helpers (`Autoupdate`, `Updater.app`, the two XPC services) with `--options runtime --timestamp --preserve-metadata=entitlements`, then the framework itself. (Alternative: leave Sparkle's shipped signature intact ‚Äî it arrives validly signed and hardened from the Sparkle project ‚Äî and only re-sign if `codesign --verify --strict` or notarization complains. Pick whichever passes; document the choice.)
3. The main app last, with `--options runtime --timestamp --entitlements script/ASTRA.entitlements`.

Keep the current `--deep` behavior for the dev/ad-hoc paths ‚Äî no reason to churn the fast local loop. Note the dev path intentionally omits library validation (`docs/code-signing.md:48`); that stays.

### 1b. Resolve the Tools-under-Resources layout question ‚Äî ‚úÖ resolved: keep as-is

The 7 helper executables are staged into `Contents/Resources/ASTRA_ASTRA.bundle/Tools/`. Mach-O executables under `Resources/` are a classic notarization/Gatekeeper friction point, but the real submission's notarization ticket explicitly lists all 7 as accepted (`ticketContents` in the `notarytool log` output) with zero issues. No relocation to `Contents/MacOS/` needed; no code or test changes required here.

**‚öÝÔ∏è `push: tags: vX.Y.Z` trigger ‚Äî wired up in code (2026-07-08), but not secure until repo settings are configured.** `on.push.tags` matches `v*.*.*` (broad on purpose ‚Äî this glob dialect doesn't reliably support regex quantifiers like `[0-9]+`); the real strictness lives in the "Determine release version and build" step (fails loudly on a malformed tag rather than silently using the latest existing one), a job-level `if: github.event.created == true` guard (rejects deletions and forced tag re-pushes), a batched-tag-push guard (rejects a stale run if a newer tag already exists), and a pre-publish check right before `gh release create` that re-verifies the remote tag still exists *and* still points at the exact commit this run built (closes both the "deleted mid-run" and "re-run after force-move" races). `skip_notarization` only applies to manual `workflow_dispatch` runs ‚Äî a tag push always does the real notarize-and-publish path.

**The gap none of that closes: GitHub runs a push-triggered workflow using the copy of `release.yml` at the pushed commit, not the copy on protected main.** Every guard above lives inside this file, so someone who can push a matching tag to an unmerged commit could edit that commit's own copy of `release.yml` to remove every one of them and still get full Developer ID + Sparkle secret access ‚Äî branch protection on `main` does not cover this path at all. Closing it requires three pieces of **manual repo-settings configuration**, which I will not do myself (modifying access controls isn't something to do without the repo owner's explicit action, and for the third item, I have never had and will never request the actual credential values):

1. **Repository ruleset** targeting tag pattern `v*` (Settings ‚Üí Rules ‚Üí Rulesets ‚Üí New ruleset ‚Äî **not** "Settings ‚Üí Tags ‚Üí New rule": that older tag-protection UI was sunset in 2024 and no longer reliably applies to a new pattern). Add yourself (or the "Repository admin" role) to the bypass list first, or you'll block your own future release tags, then enable **Restrict creations**, **Restrict updates** (blocks force-moving an existing tag ‚Äî the most direct fix for the forced-tag-update risk), and **Restrict deletions**. ‚Äî **‚úÖ done, confirmed by the user (2026-07-08).**
2. **GitHub Environment** named `release` (Settings ‚Üí Environments ‚Üí New environment) ‚Äî `release.yml`'s job now references `environment: release`, so once this environment exists with (recommended) **required reviewers**, GitHub enforces that server-side regardless of what any pushed commit's workflow file says. **Critical:** the deployment policy must allow the `v*` **tag** pattern, not just the `main` **branch** ‚Äî GitHub matches deployment policies against `GITHUB_REF`, and a tag-push run's ref is `refs/tags/v0.1.27`, not `main`. A branch-only policy would silently reject every real tag-push release. Add both: the `main` branch (for `workflow_dispatch` runs) and the `v*` tag pattern (for tag-push runs). ‚Äî **‚úÖ done, confirmed by the user (2026-07-08).**
3. **Move the 7 release secrets from repository scope to environment scope ‚Äî ‚ùå still required, and the one that actually closes the gap.** Steps 1 and 2 alone don't fully close it: `environment: release` only gates *this job's progress* (approval, deployment policy) ‚Äî it does not gate secret *access*. `ASTRA_SIGN_IDENTITY` and the rest are currently repository secrets, visible to any job in any workflow run regardless of environment. Someone who can push a tag to an unmerged commit could simply delete the `environment: release` line from that commit's copy of `release.yml`, and the job would still read every repository secret ‚Äî the environment gate never applies to a job that doesn't declare it. Closing this for real:
   - Settings ‚Üí Environments ‚Üí `release` ‚Üí add all 7 as **environment secrets** (`ASTRA_SIGN_IDENTITY`, `ASTRA_DEVELOPER_ID_CERTIFICATE_P12`, `ASTRA_DEVELOPER_ID_CERTIFICATE_PASSWORD`, `ASTRA_NOTARY_API_KEY_P8`, `ASTRA_NOTARY_KEY_ID`, `ASTRA_NOTARY_ISSUER_ID`, `ASTRA_SPARKLE_PRIVATE_KEY_B64`), same values as today's repository secrets.
